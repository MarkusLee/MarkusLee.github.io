HashSet vs. TreeSet on handling “key”s

Contrary to my intuition, the two set’s contains method returned different outcomes, which is rather confusing. So I checked the implementation of HashSet , TreeSet , and Date (JDK 11), then I found the answer.

HashSet code

HashSet is implemented based on HashMap.

We can see that HashSet get a key by searching for a Node, which is an key-value pair. The key is the object’s hash code, and the value is a reference to the object. The hash code ofDate is calculated by its contents, so as we changed the content, its hash code changed as well. But a new Node which contains the new hash code and the object reference is not added to the map of HashSet. Therefore, HashSet thought that the new Date object is not in it, although the reference hadn’t changed.

TreeSet code

TreeSet is implemented based on TreeMap.

In TreeMap, we can see that it uses Entry which is based on the object’s reference instead of its hash code. So even if wo modified the Date object, the Date object’s reference d hadn’t changed, which means TreeSet can still find the object in itself.

Conclusion

Be careful when we use mutable types in HashSet. Try to avoid such circumstances.
