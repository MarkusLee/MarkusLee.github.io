On Exceptions

Motivation

在软件构造作业1中，我写出了的这样的一段代码，用于判断矩阵中每行和是否都相等，如果相等则返回每行的和，如果不全相等则抛出一个自己定义的异常并且返回。

但是当我回头看这一段代码的时候，我感觉这段代码其实是存在问题的。关键在于行和不相等情况的处理，这里其实并不需要使用异常。因为在调用该函数之前，已经确保矩阵中每个整数必须是正数，因此完全可以返回一个非正数（例如 -1 ）来表明行和不相等。使用异常反而增加了代码的复杂度以及代码的运行开销。

于是我去各处浏览了关于异常使用方面的一些观点，总结如下。

Why Use Exceptions

异常情形 (exceptional condition) 指当前环境无法获得足够信息来解决问题，与之相反，普通问题指当前环境下能得到足够的信息来处理这个问题。
因此，在异常情形下，应该把问题交给上一级环境（即抛出异常）。

Exceptions & Ordinary Control Flow

在 Effective Java 中有这样一段话

Exceptions are, as their name implies, to be used only for exceptional conditions; they should never be used for ordinary control flow. More generally, use standard, easily recognizable idioms in preference to overly clever techniques that purport to offer better performance.

因此，应该只在处理非正常情况 (unexpected cases) 的时候使用异常，而不应该将异常加入一般控制流（例如条件判断）来处理可预知的情况。
又因为抛出和捕获异常需要花费很大的代价，所以为了提高程序的效率，一种更好的方法是使用不同的返回值来表示当前方法的处理结果。

此外，作者还提到

A well-designed API must not force its clients to use exceptions for ordinary control flow.

Exceptions & Assertions

异常用来检查一个可能发生的状况，并对其进行处理，以提高程序的健壮性。如果异常出现错误，则程序进入到try-catch的异常处理，会继续运行。

  • use exceptions when checking parameters passed to public or protected methods and constructors
  • use exceptions when interacting with the user or when you expect the client code to recover from an exceptional situation
  • use exceptions to address problems that might occur

而断言应该用来检查一个不应该发生的情况，从而发现代码逻辑中的错误，以确保程序的正确性。如果断言出现错误，则程序会终止运行。并且在运行的时候，断言被默认禁用，因此断言一般只用于debugging

  • use assertions when checking pre-conditions, post-conditions and invariants of private/internal code

  • use assertions to provide feedback to yourself or your developer team

  • use assertions when checking for things that are very unlikely to happen otherwise it means that there is a serious ﬂaw in your application

  • use assertions to state things that you (supposedly) know to be true

Preventing / Catching Exception

只有当异常成为某种可行的程序状态 (valid program state) 的时候，才应该防止异常的产生（例如使用 if (foo != null) {...} 来防止 NullPointerException 的产生）
